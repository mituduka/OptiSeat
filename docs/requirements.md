## 1. プロジェクト概要

### 1.1 目的

小学校担任教員の席替え作業における以下の課題を解決する:

- **時間コスト**: 手動での席替えは制約が多く、多大な時間を要する
- **公平性の担保**: 「いつも前列にいる児童・生徒」「前回と同じ隣席」などの偏りを排除しにくい
- **特性への配慮**: 視力・集中力・人間関係など複数の条件を同時に満たすことが困難

### 1.2 解決手段

ASP（Answer Set Programming）を用いて、制約を宣言的に記述し最適な座席配置を自動生成する。

### 1.3 ターゲットユーザー

- 日本の学校教員（ITスキル: 標準的）
- 利用端末: Windows PC / Chromebook など
- 利用シーン: 学期ごとまたは月次の席替え（学期に1〜3回程度）

### 1.4 システム構成

FastAPI バックエンドが clingo を実行するセルフホスティング構成。フロントエンド（Next.js）は氏名を内部 ID に変換してからバックエンドへ送信する。技術構成の詳細は [`architecture.md`](architecture.md) を参照。

| 役割 | 説明 | 主な関心事 |

## 3. ユースケース

### UC-01: 新学期の席替え（基本シナリオ）

```
アクター: 担任教員
事前条件: 児童・生徒名簿・教室レイアウトが設定済み
事後条件: 印刷可能な座席表が生成されている

主フロー:
1. 教員が「新しい席替えを開始」を選択
2. システムが前回の配置（存在する場合）を表示
3. 教員が必要に応じて制約を追加・変更
4. 教員が「計算実行」をクリック
5. システムが複数の座席配置候補を提示（最大5案）
6. 教員が最も適した案を選択
7. 教員がドラッグ＆ドロップで微調整（任意）
8. 教員が「確定・保存」をクリック
9. システムが座席表のPDFファイルを出力

代替フロー:
3a. 前回配置が存在しない場合: 前回配置のSoft制約はスキップ
5a. 制約が矛盾して解が存在しない場合:
    5a.1 システムが「解なし」を通知（事前チェックで検出できる代表的な矛盾は条件設定時に警告表示）
    5a.2 教員が制約を緩和して再実行
```

### UC-02: 名簿の登録・更新

```
アクター: 担任教員
主フロー:
1. 教員が「名簿管理」画面を開く
2. 手動入力（氏名・性別・配慮事項）
3. 各児童・生徒に属性（性別・タグ）を設定
4. 「保存」をクリック
```

### UC-03: 教室レイアウトの設定

```
アクター: 担任教員
主フロー:
1. 教員が「教室設定」画面を開く
2. 行数・列数を入力（例: 6行×5列 = 30席）
3. 黒板側（前方）の方向を指定
4. 使用しない座席（欠席用など）をクリックして除外
5. 「保存」をクリック
```

## 4. 機能要件

### 4.1 名簿管理

#### FR-01: 名簿インポート

| 項目 | 内容 |
|------|------|
| 入力 | 手入力（氏名・性別・配慮事項） |
| 必須フィールド | 氏名（または出席番号） |
| 任意フィールド | 性別、タグ（後から追加可能） |
| 最大人数 | 50名（想定: 通常クラスの最大人数） |
| エラー条件 | 重複する出席番号・空の氏名は登録不可 |

#### FR-02: 属性（タグ）の設定

各児童・生徒に以下のタグを0個以上付与できる。タグは複数選択可能。
タグは「どこに座りたいか」の希望を表す。理由（視力・体格など）は区別しない。

| タグID | 表示名 | 説明 |
|--------|--------|------|
| `front_preferred` | 前列希望 | 前列に座りたい児童・生徒（理由不問） |
| `back_preferred` | 後列希望 | 後列に座りたい児童・生徒（理由不問） |

#### FR-03: 座席固定

- 任意の児童・生徒を任意の座席に固定できる
- 固定された座席はソルバのHard制約として扱われる
- 固定の解除はいつでも可能

### 4.2 教室レイアウト管理

#### FR-04: グリッド生成

| 項目 | 内容 |
|------|------|
| 入力 | 行数（縦）と列数（横）の数値 |
| 範囲 | 行: 1〜12、列: 1〜12（上限は環境変数 `SEAT_MAX_ROWS` / `SEAT_MAX_COLS` で変更可能） |
| 前列の指定 | 黒板側の行番号（Row 1 を前列とするなど） |
| 座席除外 | クリックで特定の座席を「空席」にできる |

#### FR-05: 前列・後列エリアの判定

グリッド設定から前列・後列エリアを判定する（前後配慮 S-01 / S-02 の判定に使用）:

| エリア | 判定基準 | 設定方法 |
|--------|---------|---------|
| `front` | ユーザーが指定した行番号に一致する行 | `classroom.front_rows` に行番号リストで指定（デフォルト: `[1]`） |
| `back` | ユーザーが指定した行番号に一致する行 | `classroom.back_rows` に行番号リストで指定（デフォルト: 最終行のみ） |

#### FR-06: 班（グループ）レイアウト設定

| 項目 | 内容 |
|------|------|
| 概要 | ユーザーが座席を任意にグループ化し「班」として登録できる |
| 設定単位 | 班ID（1始まりの整数）と、その班に属する座席IDの集合 |
| 制約 | 1つの座席は1つの班にのみ所属できる。班未所属の座席があっても可 |
| 利用目的 | 班内の男女バランス（S-03）・孤独感緩和（S-04）Soft制約に使用 |

班の「レイアウト（どの座席が何班か）」はユーザーが事前に設定する。
ソルバは「どの児童・生徒をどの座席に割り当てるか」を計算するのみで、班レイアウト自体は変更しない。

### 4.3 制約設定

制約の詳細は [セクション5](#5-制約条件の詳細仕様) を参照。

#### FR-07: Hard制約のUI

- 座席グリッド上のクリック操作で座席固定・座席性別制約・空席を設定
- 児童・生徒ペアの制約（隣接禁止・同班禁止・隣接固定）は2名を選択して登録
- 班分散グループはグループ単位でメンバー（2名以上）を登録
- 現在有効な制約の一覧を表示し、個別に削除できる

#### FR-08: Soft制約の設定

- 班内配慮（`gender_balance` / `loneliness`）は ON/OFF トグルで個別に有効化できる（既定: OFF）
- 前回座席への配慮（`differ_seat` / `differ_neighbor` / `differ_group`）も同様に ON/OFF できる
- 各制約の重みと優先度レベルは固定（[セクション5.2](#52-soft制約最適化条件) 参照）。ユーザーによる重みの変更はできない

### 4.4 ソルバ実行と結果表示

#### FR-09: ソルバ実行

| 項目 | 内容 |
|------|------|
| 入力 | 児童・生徒リスト・教室レイアウト・制約設定 |
| 出力 | 座席配置の候補リスト（既定5案・上限10案） |
| タイムアウト | 既定30秒・上限120秒（超過時はその時点までに得られた解を返す） |
| 最大計算対象 | 50名 |

#### FR-10: 複数案の提示

- ソルバは最適スコアの解を最大解数（既定5案）まで列挙する
- 各案に対してスコア内訳を表示（どのSoft制約に何件違反しているか）
- 教員はタブ切り替えで案を比較・選択できる

#### FR-11: 手動微調整

- 選択した案のグリッドビューで、座席をドラッグ＆ドロップ（モバイルはタップ交換）で入れ替え可能
- 入れ替えた結果、Hard制約に違反する場合は警告を表示
- 微調整後のスコアをリアルタイムで再計算・表示

#### FR-12: 出力・エクスポート

| 形式 | 内容 |
|------|------|
| PDF | 印刷用座席表（クライアントサイドでベクター生成。グリッド形式、氏名表示） |
| JSON | 名簿・条件・確定した配置のエクスポート/インポート（次回の席替えで再利用） |
| 画面表示 | ブラウザ上でのプレビュー |

### 4.5 前回配置の利用

#### FR-13: 前回配置の登録と利用

- 確定した配置は JSON エクスポートに含まれ、次回インポート時に「前回座席」として読み込まれる
- 前回座席は条件設定画面で手動でも登録・編集できる
- 前回座席はSoft制約（S-05〜S-07）の入力として使用される
- アプリ内に複数世代の履歴は保存しない（保持する前回配置は1世代のみ）

## 5. 制約条件の詳細仕様

### 5.1 Hard制約（必須条件）

Hard制約に違反する解は**生成されない**。

| ID | 名称 | 説明 | ASPルール種別 |
|----|------|------|--------------|
| H-01 | 1人1席 | 各児童・生徒は必ず1つの座席に割り当てられる | choice rule（`base.lp`） |
| H-02 | 1席1人 | 各座席には最大1人の児童・生徒のみ配置される | 整合性制約 |
| H-03 | 座席固定 | `fixed(S, R)` が指定された場合、児童・生徒SはR席に必ず配置 | 整合性制約 |
| H-04 | 8方向隣接禁止 | `forbidden8(S1, S2)` が指定された場合、斜め含む8方向に隣接させない | 整合性制約 |
| H-05 | 同班禁止 | `forbidden_group(S1, S2)` が指定された場合、同じ班に配置させない | 整合性制約 |
| H-06 | 座席性別制限 | `seat_gender_constraint(R, G)` が指定された場合、その座席には指定性別の児童・生徒のみ配置 | 整合性制約 |
| H-08 | 隣接固定（relative_fixed） | 児童・生徒Aの座席から相対オフセット(d_row, d_col)に児童・生徒Bを配置 | 整合性制約 |
| H-10 | 班分散グループ（leader_group） | 同一グループに登録されたメンバー同士を同じ班に配置しない | 整合性制約 |

ID は実装（`backend/solver/lp/hard.lp`）のコメント番号に対応する（H-07・H-09 は欠番）。

> [!IMPORTANT]
> 「前回と異なる座席・横隣・班」（`differ_seat` / `differ_neighbor` / `differ_group`）はHard制約ではなく、
> Soft制約（S-05・S-06・S-07）として実装されている。詳細は [セクション5.2](#52-soft制約最適化条件) を参照。

H-04/H-05 は `ForbiddenConstraint.type` で切り替える:
- `adjacent8`（デフォルト）→ H-04（8方向隣接禁止）
- `same_group` → H-05（同班禁止）

#### adjacent8 の定義

```
adjacent8（8方向）: 斜め含む8方向
  ・行差が1以下かつ列差が1以下（自分自身を除く）
```

### 5.2 Soft制約（最適化条件）

Soft制約は違反するとペナルティが加算される。ペナルティが最小の解が優先される。
班内配慮は `soft_toggles`（`gender_balance` / `loneliness`）、前回配慮は `prev_options` で
個別に有効化・無効化できる。ID は実装（`backend/solver/lp/soft.lp`）の弱制約番号に対応する。

| ID | 名称 | 重み | 優先度レベル | 有効化 | 説明 |
|----|------|------|------------|--------|------|
| S-01 | 前列希望・前列配置 | D（行距離） | @5（最高） | 常時 | `front_preferred` タグの児童・生徒が前列から離れるほどペナルティが増える |
| S-02 | 後列希望・後列配置 | D（行距離） | @5（最高） | 常時 | `back_preferred` タグの児童・生徒が後列から離れるほどペナルティが増える |
| S-03 | 班内男女バランス | 1 | @4 | `gender_balance` | 班に男女どちらか一方が0人の場合（班単位で1件カウント） |
| S-04 | 孤独感緩和（同性ピア配置） | 1 | @4 | `loneliness` | 横隣∪同班メンバーの中に同性が1人もいない場合 |
| S-05 | 前回同席 | 1 | @1（最低） | `prev_options.differ_seat` | 前回と同じ座席に配置された件数 |
| S-06 | 前回横隣ペア解消 | 1 | @3 | `prev_options.differ_neighbor` | 前回横隣だったペアが今回も横隣になっている件数 |
| S-07 | 前回同班解消 | 1 | @2 | `prev_options.differ_group` | 前回と全く同じメンバー構成の班の件数 |

> [!IMPORTANT]
> **男女に関する制約はS-03とS-04のみ**。S-04は班を跨いだ横隣も対象に含める（詳細はASP設計書参照）。
> S-01/S-02 の重みは固定値ではなく、最近接の前列/後列からの行距離 D（gradual penalty）。

#### S-04 詳細: 孤独感緩和ピア集合

各児童・生徒について「孤独感緩和ピア集合」を以下の通り定義する:

```
孤独感緩和ピア集合 = ① 横隣（左右、最大2人）∪ ② 同班の他メンバー（班未所属の場合は空）
```

- ①の横隣は**班を跨いでいても対象**（窓側・廊下側は片側のみで最大1人）
- ②の同班メンバーは横隣でなくてもよい
- この集合の中に**同性が1人以上いなければ**ペナルティを課す

#### ペナルティの優先度レベル（`@` レベル）

clingoの弱制約では `@N` でレベルを指定し、**高いレベルから優先的に最小化**される。

```
@5: 座席希望（前列・後列配慮）
@4: 班内の男女配慮（バランス・孤独感緩和）
@3: 前回横隣ペアの解消
@2: 前回班構成の変更
@1: 前回と異なる座席
```

### 5.3 制約の競合処理

Hard制約同士が競合する場合（例: 固定した2人が隣接禁止ペア）、
ソルバは解なし（UNSATISFIABLE）を返す。

- 代表的な矛盾パターン（同一座席への複数固定・固定と性別制約の矛盾・隣接固定と隣接禁止の併用など）は、
  ソルバ実行前に `/api/v1/validate` とフロントエンドの矛盾チェックで警告表示される
- 解なしとなった場合、ソルバはどの制約が競合したかまでは特定しない。教員が制約を緩和して再実行する

## 6. 非機能要件

| ID | 要件 | 基準 |

| NFR-S-03 | 通信 | HTTPS必須（本番環境） |
| NFR-S-04 | 第三者送信禁止 | 児童・生徒の情報を外部サービスに送信しない |

### 6.4 信頼性・可用性

| ID | 要件 | 内容 |
|----|------|------|
| NFR-R-01 | データバックアップ | エクスポート機能によるデータのローカル保存 |

### 6.5 互換性

| ID | 要件 | 内容 |
|----|------|------|
| NFR-C-01 | ブラウザ対応 | Chrome / Edge 最新版（Safari・Firefoxは参考対応） |

## 7. 用語定義

| 用語 | 定義 |
|------|------|
| **席替え** | 学期ごとに行われる座席の再配置作業 |
| **Hard制約** | 必ず満たされなければならない条件。違反する解は生成されない |
| **Soft制約** | できれば満たしたい条件。違反してもよいが、ペナルティが付く |
| **ASP** | Answer Set Programming。組み合わせ最適化を宣言的に記述する論理プログラミングパラダイム |
| **clingo** | Potassco Labsが開発するASPソルバ |
| **clorm** | clingoのPython ORM層。PredicateクラスでASP事実を扱う |
| **Predicate（述語）** | ASPにおける関係の基本単位。`assign(1, 5).` のような事実 |
| **FactBase** | clormが提供するPredicate集合のコンテナ。クエリ機能を持つ |
| **前列** | 黒板に最も近い行（グリッドの先頭行） |
| **隣接** | 8方向隣接（斜め含む上下左右） |
| **固定** | 特定の児童・生徒を特定の座席に固定する操作（旧称: ピン留め） |
